# TSKHook UI 简体中文翻译工作指导

本文是交给翻译大模型的强制工作规范。目标是把《Twinkle Star Knights X》的日文 UI 文本翻译为自然、准确、简洁的简体中文，同时确保翻译文件可以被程序安全读取。

## 1. 任务边界

输入文件是 JSONL：每一行都是一个独立 JSON 对象。你只负责翻译 `source`，并把结果写入同一对象的 `translation` 字段。

你只能进行以下两项修改：

1. 把 `translation` 从空字符串改为简体中文译文。
2. 把成功完成的条目的 `status` 从 `new` 或 `stale` 改为 `translated`。

不得修改、删除、重排或重新生成其他字段，包括但不限于：

- `key`
- `category`
- `identity`
- `source`
- `sourceHash`
- `previousSourceHash`
- `origins`
- `occurrences`

`translated` 只表示机器初译完成。不要把状态写成 `reviewed` 或 `verified`，这两个状态只能由人工审校和游戏内验证产生。

## 2. 输出协议

- 输出必须仍是 JSONL，一条输入对应一条输出，行数、顺序和对象数量完全一致。
- 每行只能有一个合法、紧凑的 JSON 对象；不要输出 Markdown 代码块、解释、标题、序号或空行。
- 使用 UTF-8 编码。
- 正确转义 JSON 中的双引号、反斜杠、制表符和换行。原文中的字面换行结构必须保留为合法的 `\n`。
- 即使无法确定译法，也不得删除该行。将 `translation` 保持为空、`status` 保持原值，留给后续批次处理。
- 不得把多个对象合并成数组，也不得只返回 `key` 和译文。

建议每批处理 100～200 行；长技能说明较多时每批 40～80 行。若模型上下文不足，应缩小批次，不能截断输出。

## 3. 翻译风格

- 使用中国大陆常用的简体中文，表达应符合二次元角色养成/回合制战斗游戏的 UI 习惯。
- 准确优先于文采，简洁优先于扩写。不要补充原文没有的信息，也不要删减机制、条件、对象或限制。
- 技能与装备说明使用明确、紧凑的机制描述；按钮、任务和菜单使用短句；角色资料可稍自然，但不得自由改写人物设定。
- 相同日文词在相同语境中保持一致。遇到既有术语表时，术语表的优先级高于模型偏好。
- 专有名词没有可靠统一译名时，暂时保留日文原名，不能自行创造看似正式的译名；可在批次外单独汇报候选术语，但不得在 JSONL 输出中加入说明。
- 不进行内容审查、弱化或规避；忠实翻译输入文本。

## 4. 必须原样保留的内容

以下结构必须逐字、逐个、按原顺序保留，只翻译它们之间的自然语言：

- 富文本和控制标签，例如 `<color=...>`、`</color>`、`<size=...>`、`<br>`。
- 格式占位符，例如 `{0}`、`{1:N0}`、`%s`、`%d`。
- 数值、正负号、倍率、百分号、范围和比较符，例如 `+50`、`1.5倍`、`20%`、`3～4`、`≧`。
- 机制缩写和属性缩写，例如 `HP`、`ATK`、`EX`、`CT`、`CRI`、`SPD`。
- 原文中的换行数量、分段顺序，以及用于并列效果的 `/`。
- 资源 ID、代码样式文本、URL 和文件名。

翻译前后应逐项核对：标签集合、占位符集合、数字序列和 `/` 数量必须一致。不要把半角占位符或标签改成全角字符。

## 5. 分类规则

根据 `category` 调整表达，但不得改变数据结构：

- `skill`：条件、目标、效果、持续时间和数值顺序必须严格对应。多个等级文本要使用同一句式，不能因润色造成机制差异。
- `item`：名称简短稳定；描述明确说明用途、来源或效果，不添加营销式修辞。
- `equip`：装备名称和套装词保持一致；属性缩写保留，强化阶段和数值不得改写。
- `unit`：角色名、称号、阵营等专名优先服从术语表；人物介绍可以自然，但不能改设定或语气。
- `reward`：奖励名和数量表达简洁，保持活动名、关卡名和兑换对象一致。
- `shop`：商品、货币、购买限制和刷新条件必须准确，按钮文本尽量短。
- `misc`：结合 `origins` 中的 API 与 JSON 路径判断场景；上下文仍不足时宁可留空，不要猜测。

## 6. 术语与一致性

以下缩写默认不翻译：`HP`、`ATK`、`EX`、`CT`、`CRI`、`SPD`。

通用词可采用下列基础译法；若项目术语表另有规定，以项目术语表为准：

| 日文 | 简体中文 |
|---|---|
| スキル | 技能 |
| EXスキル | EX技能 |
| ダメージ | 伤害 |
| クリティカル | 暴击 |
| クールタイム | 冷却时间 |
| 行動 | 行动 |
| 味方 | 我方 |
| 敵 | 敌人 |
| 全体 | 全体 |
| 装備 | 装备 |
| 強化 | 强化 |
| 報酬 | 奖励 |

同一 `identity` 下的名称、简介、技能等级和说明应互相参照。近似文本不要逐条自由发挥：先确定共同句式，再只翻译实际不同的部分。

## 7. 歧义处理

- `origins` 仅用于理解文本出现在哪个 API 和 JSON 路径中，不得翻译或修改。
- 短词存在多义时，优先参考同一 `key` 前缀、同一 `identity` 或相邻条目的说明。
- 如果上下文仍不足以判断，保持空译文和原状态。不要用“未知”“待确认”占据 `translation`。
- 若 `status` 是 `stale`，必须以当前 `source` 重新翻译；旧 `translation` 只能作为参考，不能直接复制后改成 `translated`。
- 若输入状态已经是 `reviewed` 或 `verified`，不得覆盖译文或降低状态，除非任务发起者明确要求重译。

## 8. 每批输出前的自检

逐行检查：

1. JSON 能否独立解析。
2. 除 `translation` 和允许变化的 `status` 外，其他字段是否与输入完全一致。
3. 行数和顺序是否与输入完全一致。
4. 是否只使用简体中文，没有无意混入繁体字。
5. 标签、占位符、数字、缩写、换行与 `/` 是否完整保留。
6. 是否存在漏译、擅自扩写、机制主客体颠倒或肯否定反转。
7. 新完成条目的状态是否恰好为 `translated`。

任何一项不通过，都应先修正再输出整批结果。

## 9. 示例

输入：

```json
{"key":"skill:skill_id=1001:lv=1:skill_detail","category":"skill","identity":{"skill_id":"1001","lv":"1"},"source":"敵全体にATKの120%のダメージ","sourceHash":"...","translation":"","status":"new","origins":[{"api":"Example","path":"$.skill_detail"}],"occurrences":1}
```

正确输出仍是一行 JSON，仅修改两个字段：

```json
{"key":"skill:skill_id=1001:lv=1:skill_detail","category":"skill","identity":{"skill_id":"1001","lv":"1"},"source":"敵全体にATKの120%のダメージ","sourceHash":"...","translation":"对全体敌人造成相当于ATK 120%的伤害","status":"translated","origins":[{"api":"Example","path":"$.skill_detail"}],"occurrences":1}
```

实际任务输出时不要包含上述代码块或任何说明，只输出处理后的 JSONL 内容。

## 10. 推荐的模型指令

可把本文件全文作为系统指令，再附加以下用户指令：

> 按照系统提供的《TSKHook UI 简体中文翻译工作指导》处理下面这批 JSONL。严格保持行数、顺序和除 translation/status 外的所有字段；只输出处理后的 JSONL，不要输出代码块或解释。无法可靠翻译的行保持 translation 为空并维持原 status。
